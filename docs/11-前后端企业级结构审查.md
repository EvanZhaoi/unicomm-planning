# 11-前后端企业级结构审查

## 1. 审查目标

本文用于记录当前 `unicomm-desktop` 与 `unicomm-server` 的真实代码结构，并约束后续开发继续按企业级分区推进。

目标不是为了拆目录而拆目录，而是保证：

- 功能边界清楚。
- 基础设施与业务模块分离。
- UI、状态、API、桌面能力不互相穿透。
- 后端 Controller、Service、实时推送、认证、数据访问职责明确。
- 文档描述与当前代码一致，避免后续开发按过期方案实现。

## 2. 前端当前结构

当前 `unicomm-desktop/src` 已形成以下主要分区：

```text
src/
├── components/              # 通用 UI 和布局组件
│   ├── layout/              # AppLayout 等应用壳层
│   └── ui/                  # shadcn/radix 风格基础组件
├── core/
│   └── http/                # HTTP 基础设施、错误体系、拦截器
├── desktop/                 # Tauri 桌面能力封装
│   ├── device/              # 设备信息
│   ├── notification/        # 系统通知
│   ├── shortcut/            # 全局快捷键
│   ├── tray/                # 托盘
│   ├── user/                # Windows 用户识别
│   └── window/              # 窗口管理
├── features/                # 业务功能模块
│   ├── auth/                # 桌面认证、设备验证码、认证状态
│   ├── memo/                # Memo 工作台、API、Store、编辑器
│   ├── notify/              # 通知中心与系统通知入口
│   └── settings/            # 设置界面
├── hooks/                   # 跨功能通用 React Hook
├── i18n/                    # 中日文语言包和翻译 Hook
├── native/                  # Tauri invoke 抽象
├── services/                # 跨功能服务，当前包含 realtime、storage 等
├── stores/                  # 全局 Store
├── styles/                  # 主题与全局样式
└── utils/                   # 通用工具
```

### 2.1 前端分区评价

当前前端分区基本合理，已经达到继续扩展的基础要求：

- `core/http` 承担请求基础设施，避免每个功能重复封装 HTTP。
- `desktop/*` 专门封装 Tauri 能力，业务组件不直接关心 Rust 命令细节。
- `features/*` 按业务域拆分，Memo、认证、设置、通知互相独立。
- `components/ui` 作为基础组件层，可以持续沉淀可复用组件。

需要继续改进的点：

- 旧版 `services/request.ts`、`services/cache.ts`、`services/storage.ts`、`services/index.ts` 和 `stores/auth.store.ts` 已清理，HTTP 与认证状态边界统一到 `core/http` 和 `features/auth/store/authStore.ts`。
- Memo 工作台组件较大，后续可拆为 `MemoListPanel`、`MemoEditorPanel`、`MemoMetadataBar`，降低单文件维护压力；右键菜单等交互优先复用 `components/ui` 基础组件。
- `notify` 当前用于通知中心和系统通知承接，通知数据先通过 Zustand persist 本地持久化，通知中心列表已使用 TanStack Virtual；后续做企业通知平台时应拆出 API、Store、类型和通用通知规则。

## 3. 后端当前结构

当前 `unicomm-server/src/main/java/com/unicomm` 采用单体模块化结构：

```text
com.unicomm/
├── common/                  # 统一返回、异常、分页、错误码
├── config/                  # CORS、数据源、OpenAPI、Sa-Token、WebSocket 配置
├── module/
│   ├── auth/                # 认证、设备信任、人员适配器、审计
│   │   ├── controller/
│   │   ├── dto/
│   │   ├── entity/
│   │   ├── integration/
│   │   └── service/
│   ├── member/              # 人员搜索接口
│   │   ├── controller/
│   │   └── dto/
│   └── memo/                # Memo 业务模块
│       ├── controller/
│       ├── dto/
│       ├── realtime/
│       └── service/
└── UniCommApplication.java
```

### 3.1 后端分区评价

当前后端分区总体清楚：

- `common` 作为跨模块基础设施。
- `config` 只放框架配置。
- `module/auth` 独立处理认证、人员来源、设备信任。
- `module/member` 对前端提供人员搜索能力。
- `module/memo` 管理 Memo CRUD、分组、相关人权限、收藏、置顶和实时事件。

需要继续改进的点：

- `MyBatisMemoService` 负责 Memo 业务编排、权限入口、响应补全和通知发布；Memo 主表、分组、相关人关系、收藏和置顶的数据访问已收敛到 Entity / Mapper。
- `module/memo/realtime` 当前是单实例内存 WebSocket 会话，后续多实例部署需要 Redis Pub/Sub 或 MQ。
- `module/auth/entity` 与 `module/memo/entity` 已作为 MyBatis-Plus 表实体使用，后续新增表应继续同步补齐 entity / mapper。
- 认证上下文仍需要继续贯彻到全部业务接口，避免任何默认用户名兜底。

## 4. 企业级分层规范

### 4.1 前端规范

| 层级 | 目录 | 允许做什么 | 不允许做什么 |
|------|------|------------|--------------|
| UI 基础层 | `components/ui` | Button、Select、Popover 等无业务组件 | 调 API、读业务 Store |
| 布局层 | `components/layout` | 应用壳、导航、窗口标题栏 | 写具体业务流程 |
| 桌面能力层 | `desktop/*` | 封装 Tauri API、系统能力 | 渲染业务 UI |
| HTTP 基础层 | `core/http` | baseURL、token、错误、拦截器 | 写 Memo/Notify 业务 |
| 通用 Hook 层 | `hooks/*` | 无业务依赖的生命周期、页面级快捷键等复用 Hook | 直接调用业务 API、读取业务 Store |
| 功能模块层 | `features/*` | API、Store、组件、类型 | 直接调用 Rust 命令细节 |
| 跨功能服务 | `services/*` | realtime、缓存、存储等应用服务 | 变成业务万能工具箱 |

### 4.2 后端规范

| 层级 | 目录 | 允许做什么 | 不允许做什么 |
|------|------|------------|--------------|
| Controller | `module/*/controller` | 参数接收、认证入口、调用 Service | 写 SQL、写复杂业务规则 |
| DTO | `module/*/dto` | 请求/响应模型 | 写业务逻辑 |
| Service | `module/*/service` | 事务、权限、业务编排 | 直接暴露给前端 |
| Realtime | `module/*/realtime` | WebSocket 事件和连接管理 | 查询和修改业务数据 |
| Integration | `module/*/integration` | 外部系统适配器 | 写业务判断 |
| Common | `common` | 通用响应、错误、分页 | 放具体业务代码 |
| Config | `config` | 框架配置 | 放业务流程 |

## 5. 当前技术栈确认

### 5.1 前端

- Tauri 2
- React 19
- TypeScript 5
- Vite 6
- Tailwind CSS 4
- Zustand 5
- TanStack Query
- TanStack Virtual
- Axios
- Radix UI + shadcn 风格自定义基础组件
- Milkdown Crepe
- Mermaid / Milkdown diagram
- Tauri notification / global shortcut

TanStack Query 已作为服务端缓存层接入，Memo store 通过 `queryClient.fetchQuery` 复用请求与缓存。Zustand 继续负责 UI 状态、编辑草稿和局部更新，避免一次性改动破坏现有功能。

TanStack Virtual 已用于通知中心列表，后续 Memo 列表、成员搜索结果和企业通知平台列表也应优先复用该依赖，避免继续手写滚动容器和全量 DOM 渲染。

Radix UI 已用于 Select、Dropdown、Popover、Tabs、Context Menu、Alert Dialog 等基础交互组件。Memo 列表右键菜单已从手写坐标菜单迁移到 `components/ui/context-menu`，删除 Memo 和删除分组已从浏览器原生确认框迁移到 `components/ui/alert-dialog`。后续同类浮层和危险操作确认优先复用基础组件，避免重复处理焦点、关闭、边界和可访问性。

`AppLayout` 的导航项、状态筛选、分组筛选、设置/通知/主题切换按钮已统一使用 `components/ui/Button`。布局层后续新增按钮应继续复用基础 Button，避免不同区域的 hover、focus、禁用和尺寸规则分裂。

`AuthStatusView` 的设备验证码输入和提交按钮已统一使用 `components/ui/Input` 与 `components/ui/Button`，认证入口文案已纳入 i18n。认证、设备信任、验证码等进入应用前的页面也必须遵守主界面组件和国际化规范。

`MemoGroupManager` 的图标选择、预设颜色选择和自定义颜色输入已统一复用 `components/ui/Button` 与 `components/ui/Input`。后续分组、标签、成员等弹层中的选择器应继续复用基础组件，避免重新手写按钮网格。

`RemoteMultiSelect` 的搜索输入和默认移除按钮已统一复用 `components/ui/Input` 与 `components/ui/Button`。相关人、成员、未来标签等远程搜索多选场景应复用该组件，不要在业务组件里重复实现搜索防抖、弹出层和胶囊项交互。

`MemoWorkspace` 的 Memo 列表项、状态切换、相关人权限切换和相关人移除按钮已统一复用 `components/ui/Button`。Memo 主工作区后续新增可点击控件应继续走基础组件，特殊触发器也应通过 `asChild` 或显式 class 保持基础行为一致。

Memo 编辑页底部保存状态已拆为 `features/memo/components/MemoSaveStatusIndicator.tsx`，左侧列表区域已拆为 `features/memo/components/MemoListPanel.tsx`，标题/分组/状态/更新时间/模式切换已拆为 `features/memo/components/MemoEditorHeader.tsx`，人员搜索和读写权限操作已拆为 `features/memo/components/MemoRelatedUsersEditor.tsx`。列表空状态、权限徽标和日期/状态格式化分别沉淀到独立组件或工具函数。后续继续拆分底部操作栏，避免 `MemoWorkspace` 继续承载过多展示细节。

页面级键盘快捷键统一通过 `hooks/useDocumentShortcut` 注册。Memo 编辑页 `Ctrl/Cmd + S` 已迁移到该 Hook，后续新增同类快捷键时业务组件只声明触发规则和回调，不再直接手写 `window.addEventListener("keydown", ...)`。

实时事件处理规则：

- WebSocket 事件只作为“数据已变化”的信号。
- 收到 Memo 或分组事件后，必须先失效对应 TanStack Query 缓存。
- 再由 Memo store 重新拉取列表、详情或分组，避免 staleTime 内读到旧缓存。

### 5.2 后端

- Java 21
- Spring Boot 4.0.6
- Spring MVC
- Spring WebSocket
- MyBatis-Plus
- Redis
- MySQL
- Sa-Token
- SpringDoc OpenAPI

MyBatis-Plus 和 Redis 已进入基础设施接入阶段。当前策略是统一 Mapper 边界：用户快照、认证审计、设备信任、设备验证码、Memo 主表、分组、相关人关系、收藏和置顶都已迁移到 MyBatis-Plus Mapper；Memo 主列表、可见性权限和聚合查询保留在 Mapper 显式 SQL 中。Redis 依赖和配置已加入，业务层默认不强制依赖 Redis 启动，当前仅预留最近认证审计缓存，后续再用于缓存、分布式会话或 WebSocket 多实例转发。

## 6. 注释规范

代码注释应围绕“为什么这样做”和“边界是什么”，避免解释显而易见的语句。

推荐注释位置：

- 认证状态机、设备信任、验证码并发控制。
- Memo 权限、相关人、收藏/置顶个人状态。
- 列表排序、分页、请求复用、实时刷新策略。
- WebSocket 连接认证、广播过滤、重连策略。
- Tauri 多窗口、系统通知点击跳转、全局右键禁用。

不推荐注释：

- 简单 getter/setter。
- JSX 中显而易见的按钮、图标、布局。
- 和代码不一致的历史方案。

## 7. 本次审查结论

当前前后端结构已经具备企业级项目的基本分层，主要问题不是目录完全错误，而是：

- 文档中仍有部分历史技术栈描述与当前实现不一致。
- 旧 HTTP、缓存、存储和认证 re-export 文件已完成清理。
- Memo 和认证关键路径需要持续补充解释性注释。
- 后续功能扩展前，应优先拆小 `MemoWorkspace` 和 `MyBatisMemoService`，避免它们继续膨胀。
