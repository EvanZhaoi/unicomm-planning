# 08-认证架构设计

## 1. 设计目标

UniComm 当前阶段采用“桌面端设备信任 + 后端用户身份上下文”的认证方案，目标是在不接入真实企业统一身份系统的测试阶段，也能提前建立接近企业级应用的认证边界。

认证设计需要满足以下目标：

- 桌面端启动后能够识别当前 Windows 用户，并向后端建立用户身份。
- 同一用户的常用设备可以被信任，避免每次打开应用都要求验证码。
- 第一次使用且没有绑定过设备时，直接绑定当前设备。
- 当同一用户从新设备登录时，要求验证码校验。
- 后续可以平滑接入企业人员 API、统一登录、邮箱验证码、审计日志和权限系统。
- Memo、通知、相关人、读写权限等业务功能都应从统一认证上下文中读取当前用户，而不是继续使用默认用户名。

## 2. 当前认证边界

当前项目分为三个主要部分：

- `unicomm-desktop`：Tauri 桌面端，负责采集本机用户和设备信息，发起认证请求，保存前端认证状态。
- `unicomm-server`：Spring Boot 后端，负责设备信任、验证码校验、用户上下文、业务权限校验。
- `unicomm-planning`：需求、设计、数据库、API 和后续实施方案文档。

当前认证方案不是最终企业单点登录方案，而是测试阶段的本地认证过渡方案。它的价值是先把“谁在使用系统”“这台设备是否可信”“业务数据应该按哪个用户过滤和授权”这些核心问题固定下来。

## 3. 认证流程

### 3.1 桌面端启动认证

桌面端打开主窗口后执行认证初始化：

1. 读取当前 Windows 用户信息。
2. 生成或读取当前设备标识。
3. 调用后端桌面认证接口。
4. 后端判断当前用户和设备是否可信。
5. 如果设备可信，直接返回认证成功。
6. 如果是该用户第一次绑定设备，后端直接信任当前设备并返回认证成功。
7. 如果该用户已有其他可信设备，当前设备未被信任，则返回需要验证码。
8. 用户输入验证码后调用验证接口。
9. 验证成功后绑定当前设备，并进入系统。

隐藏的简易 Memo 窗口不能在应用启动时主动发起认证，否则会导致主窗口和隐藏窗口同时请求认证。简易窗口只应在真正显示或获得焦点时初始化认证。

### 3.2 首次设备绑定

首次设备绑定规则：

- 如果用户从未绑定过可信设备，则当前设备自动绑定。
- 首次绑定不需要验证码。
- 只有该用户已经存在可信设备，并且当前设备与可信设备不一致时，才要求验证码。

后端需要对同一用户的设备认证流程做并发保护。桌面端可能因为窗口、插件或生命周期事件产生并发请求，如果没有按用户加锁，可能出现第一次请求刚绑定设备、第二次请求误判为新设备并要求验证码的问题。

### 3.3 新设备验证码

当用户已有可信设备，但当前设备未被信任时：

1. 后端生成设备验证码。
2. 测试阶段验证码可打印在后端日志中。
3. 生产阶段应通过企业邮箱、短信、企业微信、钉钉或统一身份平台发送验证码。
4. 用户在桌面端输入验证码。
5. 后端校验验证码是否正确、是否过期、是否已使用。
6. 校验通过后将当前设备加入可信设备列表。

邮件发送部分当前阶段可以保留 `TODO`，但接口和数据模型需要提前按生产方案设计。

## 4. 技术方案

### 4.1 桌面端技术

桌面端使用：

- Tauri：提供 Windows 桌面应用容器、系统通知、窗口控制、系统信息读取能力。
- React：构建主界面、简易 Memo 窗口、设置界面和通知界面。
- TypeScript：保证前端认证状态、接口返回值和业务模型类型安全。
- Web API / Tauri API：读取窗口状态、控制窗口显示、处理通知点击和深链跳转。

桌面端需要保存的认证状态包括：

- 当前用户姓名或用户名。
- 当前用户工号。
- 当前设备标识。
- 当前设备是否可信。
- 是否需要验证码。
- 验证码请求 ID 或临时认证会话 ID。

桌面端不应该长期保存明文验证码、密码或敏感令牌。后续如接入真实 token，需要使用 Tauri 安全存储或系统凭据管理能力。

### 4.2 后端技术

后端使用：

- Spring Boot：提供认证接口、业务 API、WebSocket 和通知接口。
- Spring MVC：承载桌面认证、验证码校验、人员搜索、Memo 权限接口。
- MyBatis-Plus + Spring JDBC / NamedParameterJdbcTemplate：认证审计、用户快照、设备信任、设备验证码等认证写入已迁移到 MyBatis-Plus；Memo 分组已迁移到 Mapper；Memo 复杂权限、分页、相关人和个人状态聚合查询暂时保留手写 SQL，按模块渐进重构。
- MySQL：持久化用户、设备信任、验证码、Memo 权限、收藏、置顶和通知数据。
- WebSocket：向相关用户推送 Memo 更新和后续通知事件。

后端认证服务需要拆分为以下职责：

- `DesktopAuthService`：处理桌面端认证入口。
- `DeviceTrustService`：处理设备信任、首次绑定、新设备验证码。
- `VerificationCodeService`：处理验证码生成、发送、校验和失效。
- `CurrentUserService`：解析当前用户上下文，供业务模块使用。
- `AuthAuditService`：记录认证成功、失败、新设备、验证码等审计事件。

### 4.3 数据库方案

认证相关建议表：

- `users`：用户基础信息，测试阶段可使用本地用户数据，生产阶段对接企业人员 API。
- `trusted_devices`：用户可信设备表，记录用户与设备的绑定关系。
- `device_verifications`：设备验证码表，记录验证码、过期时间、使用状态。
- `auth_audit_logs`：认证审计表，记录认证请求、验证结果、设备变化。

设备信任表建议字段：

- `id`
- `username`
- `employee_no`
- `device_id`
- `device_name`
- `os_name`
- `os_version`
- `trusted_at`
- `last_seen_at`
- `disabled`
- `created_at`
- `updated_at`

验证码表建议字段：

- `id`
- `username`
- `employee_no`
- `device_id`
- `code_hash`
- `expire_at`
- `used_at`
- `request_ip`
- `created_at`

验证码不应以明文长期保存。测试阶段可以为了调试打印验证码，但数据库中建议保存 hash。

## 5. 用户身份来源

### 5.1 当前测试阶段

当前测试阶段可以通过 Windows 当前用户、本地配置或后端测试数据建立用户身份。

需要避免继续在业务代码中硬编码默认用户名，例如 `Evan.Zhao`。所有业务接口都应从认证上下文中读取当前用户。

当前用户信息至少需要包括：

- 用户名
- 姓名
- 工号
- 部门
- 邮箱
- 账号状态

### 5.2 后续真实人员 API 接入

后续接入企业真实人员 API 时，建议采用适配器模式：

- `UserDirectoryClient`：定义统一人员查询接口。
- `MockUserDirectoryClient`：测试阶段使用本地或数据库模拟数据。
- `EnterpriseUserDirectoryClient`：生产阶段调用真实企业人员 API。

真实人员 API 需要支持：

- 根据工号查询人员。
- 根据姓名或工号模糊搜索人员。
- 查询用户部门、邮箱、状态。
- 判断用户是否离职或禁用。
- 批量查询相关人信息。

业务代码只依赖统一的 `UserDirectoryClient`，不直接依赖具体企业 API。这样以后切换人员来源时，不需要大范围改业务代码。

## 6. 权限设计

### 6.1 当前用户上下文

后端每个业务请求都应先解析当前用户上下文：

- 当前用户是谁。
- 当前用户是否已认证。
- 当前设备是否可信。
- 当前用户是否有权访问目标数据。

Memo 相关接口不能只依赖前端传入的用户名，也不能使用固定默认用户名。前端传入的数据只能作为请求参数，最终权限判断必须以后端认证上下文为准。

### 6.2 Memo 权限

Memo 权限分为：

- 创建人：拥有完整读写权限。
- 相关人-可读：可以查看 Memo，但不能修改正文、标题、状态、分组等核心内容。
- 相关人-可写：可以查看和编辑 Memo。

收藏、置顶应按“用户 + Memo”维度存储。因为同一个 Memo 可能被多人查看，收藏和置顶是个人行为，不能直接存到 Memo 主表。

### 6.3 与我相关

“与我相关”模块展示当前用户被加入相关人的 Memo。

该模块规则：

- 不显示新建按钮。
- 不显示或不允许修改创建人的个人分组。
- 只展示当前用户有查看权限的 Memo。
- 根据当前用户在相关人表中的权限决定是否允许编辑。

## 7. WebSocket 与通知

Memo 多人协作后，需要通过 WebSocket 实时展示最新数据。

### 7.1 Memo 更新通知

当其他人更新了当前用户可见的 Memo 时：

- 如果更新者是自己，不需要通知自己。
- 如果当前用户是相关人，需要收到更新通知。
- 通知内容建议包含 Memo 标题、更新人、更新时间和摘要。
- 点击系统通知后，应打开桌面应用并跳转到指定 Memo。

### 7.2 通知平台预留

通知分两类：

- Memo 通知：由 Memo 模块产生，主要服务协作编辑。
- 通用通知：提供后端接口，允许其他系统向桌面端发送通知。

通用通知后续需要保存到数据库，具备企业通知平台能力：

- 通知创建接口。
- 指定接收人或部门。
- 通知级别。
- 已读/未读状态。
- 桌面系统通知。
- 通知中心列表。
- 点击通知后的业务跳转。

## 8. 安全与审计

### 8.1 安全原则

- 不在前端硬编码用户身份。
- 不在数据库长期保存明文验证码。
- 不信任前端传入的用户名作为最终权限依据。
- 新设备登录需要验证码。
- 用户禁用或离职后，应阻止继续认证。
- 设备丢失时，应允许后台禁用该设备。

### 8.2 审计事件

建议记录以下事件：

- 桌面认证请求。
- 首次设备绑定。
- 新设备验证码生成。
- 验证码校验成功。
- 验证码校验失败。
- 设备禁用。
- 用户认证失败。
- 权限拒绝访问。

审计日志用于后续排查异常登录、权限问题和企业安全合规。

## 9. 接口设计建议

认证接口建议：

- `POST /api/auth/desktop/verify`：桌面端认证。
- `POST /api/auth/desktop/verification/confirm`：验证码确认。
- `GET /api/auth/me`：获取当前认证用户。
- `POST /api/auth/device/trust`：绑定可信设备。
- `POST /api/auth/device/disable`：禁用可信设备。

人员接口建议：

- `GET /api/users/search?keyword=`：按姓名或工号搜索人员。
- `GET /api/users/{employeeNo}`：查询人员详情。
- `POST /api/users/batch`：批量查询人员。

通知接口建议：

- `POST /api/notifications/test`：测试系统通知。
- `GET /api/notifications`：查询通知列表。
- `POST /api/notifications/{id}/read`：标记已读。
- `POST /api/notifications/external`：外部系统发送通知。

### 10.1 已完成方向

- 桌面端认证入口。
- 当前 Windows 用户识别。
- 设备信任服务。
- 首次设备自动绑定规则。
- 新设备验证码流程雏形。

- 生产级验证码发送能力。

## 11. 推荐实施顺序

1. 完成当前用户上下文改造，移除业务接口中的默认用户名。
2. 完成设备信任表和验证码表的稳定实现。
3. 完成 Memo 创建人、相关人读写权限的后端校验。
4. 完成前端根据权限展示只读或可编辑状态。
5. 完成 WebSocket 中的用户认证和 Memo 更新推送。
6. 完成系统通知弹窗和点击跳转。
7. 接入真实人员 API。
8. 增加设备管理和认证审计界面。
9. 扩展为通用企业通知平台。
