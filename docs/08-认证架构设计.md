# UniComm 桌面端认证架构设计

## 一、认证目标

**核心目标：** 实现"免登录"体验 + 后端身份校验

UniComm 不做传统登录页面。用户不需要在应用内输入账号密码。

但这**不是无认证**，而是：
- 桌面端读取 Windows 当前登录用户信息
- 后端校验用户是否为公司员工、是否启用
- 后端签发 Session Token
- 后续 API 请求携带 Token

---

## 二、认证边界

| 场景 | 处理方式 |
|------|----------|
| 用户是公司员工且启用 | 认证成功，建立 Session |
| 用户不在公司人员系统中 | 认证失败，显示"未授权" |
| 用户已停用 | 认证失败，显示"用户已停用" |
| 后端服务不可用 | 进入离线模式或重试 |
| 设备未授权（Phase 2） | 记录设备，后续可选绑定 |

## 四、桌面端采集信息

### 4.1 Windows 用户信息

通过 Tauri Rust 命令 `get_current_windows_user()` 获取：

| 字段 | 类型 | 说明 | 来源 |
|------|------|------|------|
| username | string | Windows 用户名 | `whoami::username()` |
| domain | string? | Windows 域（可为 null） | 环境变量或 AD |
| computerName | string | 计算机名 | `whoami::devicename()` |
| os | string | 操作系统 | `std::env::consts::OS` |
| osVersion | string | 系统版本 | 系统 API 或 fallback |

### 4.2 设备信息

通过 `DeviceService` 获取：

| 字段 | 类型 | 说明 |
|------|------|------|
| deviceId | string | 设备唯一标识（UUID，首次生成后持久化） |
| computerName | string | 计算机名 |
| os | string | Windows / macOS / Linux |
| osVersion | string | 系统版本字符串 |
| appVersion | string | UniComm 应用版本 |

### 4.3 非 Windows 环境降级

| 环境 | 处理方式 |
|------|----------|
| macOS 开发环境 | 使用 `whoami` 获取用户名，computerName 使用 hostname |
| Linux 开发环境 | 使用 `whoami` 获取用户名 |
| Windows | 正常读取 AD 用户信息 |

### 4.4 Domain 为 null 的处理

- 后端 `AuthServiceImpl.buildUserKey()` 处理 null 情况
- Key 格式：`domain#username`（domain 为空时 prefix 为空字符串）
- Phase 1 临时员工数据支持 null domain

## 五、后端 verify 流程

### 5.1 接口定义

**端点：** `POST /api/v1/auth/desktop/verify`

**请求参数：**

```json
{
  "username": "evan.zhao",
  "domain": "COMPANY",
  "computerName": "CN-SH-001",
  "deviceId": "uuid-xxx",
  "os": "Windows",
  "osVersion": "Windows 11",
  "appVersion": "0.1.0"
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| username | string | 是 | Windows 用户名 |
| domain | string | 否 | Windows 域信息 |
| computerName | string | 是 | 计算机名称 |
| deviceId | string | 是 | 客户端设备标识 |
| os | string | 是 | 操作系统 |
| osVersion | string | 是 | 操作系统版本 |
| appVersion | string | 是 | 应用版本 |

**成功响应：**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "username": "evan.zhao",
    "employeeNo": "E10001",
    "displayName": "Evan Zhao",
    "departmentName": "IT Department",
    "email": "evan.zhao@company.com",
    "permissions": ["memo:read", "memo:write"],
    "accessToken": "server-issued-token"
  }
}
```

**错误响应：**

| code | message | 说明 |
|------|---------|------|
| 401 | "用户不存在" | username 不在公司人员系统中 |
| 403 | "用户已停用" | 员工状态为 inactive |
| 500 | "认证服务异常" | 服务端错误 |

### 5.2 后端处理流程

```
接收请求
    │
    ▼
校验参数完整性
    │
    ▼
查询人员适配器 (PersonnelProvider)
    │
    ├─── 员工不存在 ────→ 返回 401
    │
    ├─── 员工停用 ────→ 返回 403
    │
    └─── 员工正常 ────→
            │
            ▼
        签发 Sa-Token Token
        (使用 username 作为 loginId)
            │
            ▼
        保存用户快照/认证审计
            │
            ▼
        校验 DeviceTrust
            │
            ├─── 常用设备 ─────→ 签发 Token
            │
            └─── 陌生设备 ─────→ 生成验证码，等待用户输入
            │
            ▼
        返回 accessToken + 用户信息
```

## 六、公司人员接口适配

### 6.1 Phase 1: 测试适配器

测试阶段不接入真实公司人员 API。当前后端已抽象 `PersonnelProvider`，并使用内存 Map 作为 `MockPersonnelProvider`，只满足本机联调、权限验证和相关人搜索，不作为生产人员主数据。

当前实现位置：
- `unicomm-server/src/main/java/com/unicomm/module/auth/integration/PersonnelProvider.java`
- `unicomm-server/src/main/java/com/unicomm/module/auth/integration/EmployeeInfo.java`
- `unicomm-server/src/main/java/com/unicomm/module/auth/integration/MockPersonnelProvider.java`
- `unicomm-server/src/main/java/com/unicomm/module/auth/service/impl/AuthServiceImpl.java`

```java
// MockPersonnelProvider.java
private final Map<String, EmployeeInfo> employeeCache = new ConcurrentHashMap<>();

@PostConstruct
public void initSeedData() {
    // 模拟公司人员接口返回
    employeeCache.put("COMPANY#evan.zhao",
        new EmployeeInfo("evan.zhao", "E10001", "Evan Zhao",
            "IT Department", "evan.zhao@company.com", "active"));
    employeeCache.put("COMPANY#alice.wang",
        new EmployeeInfo("alice.wang", "E10002", "Alice Wang",
            "HR Department", "alice.wang@company.com", "active"));
    employeeCache.put("COMPANY#bob.li",
        new EmployeeInfo("bob.li", "E10003", "Bob Li",
            "IT Department", "bob.li@company.com", "inactive"));
}
```

**种子用户说明：**
- Evan Zhao: 正常员工，可登录
- Alice Wang: 正常员工，可登录
- Bob Li: 已停用，不可登录
- Leader Wang / PM Li: 正常员工，用于相关人搜索和共享权限测试

**测试阶段约束：**
- 不调用 LDAP、AD、HR、OA 或统一身份平台的真实 API
- 不在本地保存真实员工主数据
- 相关人搜索使用同一份临时人员缓存
- 权限判断仍以后端 Token 中的 loginId 为准，不能信任前端传入的 username、owner 或 permission

### 6.2 Phase 2: 真实人员接口接入方案

人员查询已经抽象为统一接口。未来接入真实人员系统时，只需要新增真实实现，并用配置选择测试实现或真实实现：

```java
public interface PersonnelProvider {
    /**
     * 根据 Windows 域和用户名查询员工信息。
     */
    EmployeeInfo findByWindowsAccount(String domain, String username);

    /**
     * 按姓名、用户名、工号或邮箱模糊搜索员工。
     */
    List<EmployeeInfo> searchMembers(String keyword, int limit);
}
```

建议实现：

| 实现 | 用途 | 数据源 |
|------|------|--------|
| `MockPersonnelProvider` | 开发、测试、演示 | 内存种子数据或测试配置 |
| `HttpPersonnelProvider` | 生产 | HR/OA/统一身份平台 HTTP API |
| `LdapPersonnelProvider` | 可选 | LDAP / AD |

建议配置：

```yaml
unicomm:
  personnel:
    provider: mock # mock / http / ldap
    base-url: https://personnel.company.com
    timeout-ms: 3000
    app-key: ${PERSONNEL_APP_KEY:}
    app-secret: ${PERSONNEL_APP_SECRET:}
    cache-ttl-seconds: 300
```

真实人员接口建议字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| username | 是 | Windows 用户名，作为 Sa-Token loginId |
| domain | 否 | Windows 域，用于区分同名账号 |
| employeeNo | 是 | 员工工号 |
| displayName | 是 | 员工显示名称 |
| departmentName | 否 | 部门名称 |
| email | 否 | 邮箱 |
| status | 是 | `active` / `inactive` / `resigned` |
| roles | 否 | 未来角色权限扩展 |

错误映射：

| 人员接口结果 | UniComm 响应 |
|--------------|--------------|
| 查询不到员工 | 401 `USER_NOT_FOUND` |
| 员工停用/离职 | 403 `USER_DISABLED` |
| 人员接口超时 | 503 `SERVICE_UNAVAILABLE`，前端显示认证服务不可用 |
| 人员接口 5xx | 503 `SERVICE_UNAVAILABLE`，记录服务端日志 |

接入顺序建议：
1. 新增 `HttpPersonnelProvider`，按公司真实 API 做字段映射和错误转换
2. 增加 provider 配置开关，在测试环境使用 `mock`，生产环境使用 `http` 或 `ldap`
3. 增加短 TTL 缓存，降低人员搜索和认证的接口压力
4. 接入用户快照与认证审计，只保存缓存和审计数据，不作为人员主数据
5. 根据公司安全要求增加接口签名、IP 白名单、服务账号密钥轮换

安全要求：
- 后端不能只根据前端传入的 username 判定身份，必须调用人员适配器校验
- 生产环境必须禁用 mock 种子用户，避免测试账号进入正式环境
- accessToken、人员接口密钥、员工隐私字段不允许写入普通日志
- 相关人搜索需要限制 `limit`、做关键词长度校验，并记录异常调用
- WebSocket 握手也需要校验 Sa-Token，后续事件只推送给创建者和相关人

## 七、Token 与 Session

### 7.1 Token 签发

使用 Sa-Token 的 `StpUtil.login(loginId)` 签发 Token：

```java
// 使用 username 作为 loginId（非 userId）
StpUtil.login(emp.username);
String token = StpUtil.getTokenValue();
```

### 7.2 Token 校验

后续 API 通过 Sa-Token 中间件自动校验：
- 请求 Header: `unicomm-token: {token}`；桌面端也会兼容携带 `Authorization: Bearer {token}`
- Sa-Token 自动校验 Token 有效性
- 自动管理 Session 过期
- Memo 模块不再使用开发期默认用户名兜底，当前用户必须来自 Token loginId

### 7.2.1 WebSocket Token 校验

Memo 实时通道同样需要认证。桌面端连接 `ws://host/ws?token={accessToken}`，后端在握手阶段通过 Sa-Token 校验 token，并把 token 对应的 loginId 写入 WebSocket session。

事件推送规则：
- 未携带 token 或 token 无效时拒绝建立 WebSocket 连接
- 服务端根据连接上的 username 与事件 `recipientUsernames` 做匹配，只给创建者和相关人推送
- 前端收到事件后仍保留本地用户判断作为兜底，但不再依赖前端过滤作为唯一权限边界
- WebSocket 只推事件，不推完整 Memo 数据；最终可见数据仍通过 HTTP API 权限查询获得

### 7.3 Token 配置 (application.yml)

```yaml
sa-token:
  token-name:unicomm-token
  timeout:259200        # 3天过期
  active-timeout:1800   # 30分钟无操作过期
  is-concurrent:false   # 不允许多端同时在线
  is-share:false        # 不跨端共享 Token
```

### 7.4 安全原则

| 禁止 | 说明 |
|------|------|
| ❌ 日志打印 accessToken | 安全风险 |
| ❌ 硬编码真实用户 | 测试阶段只能使用 mock 人员，生产必须禁用 |
| ❌ 前端直接信任 username | 必须后端校验 |

## 八、用户快照/认证审计

### 8.1 设计目的

| 项目 | 说明 |
|------|------|
| 不是用户主表 | 人员真实信息以公司人员接口为准 |
| 用于缓存 | 减少频繁调用公司人员接口 |

## 九、设备信任 DeviceTrust

### 9.1 设计目的

| 项目 | 说明 |
|------|------|
| 安全审计 | 记录设备访问历史 |
| 常用设备判断 | 当前用户使用已信任设备时直接进入 |
| 陌生设备验证 | 当前设备不在信任列表时，生成验证码并要求用户输入 |

### 9.2 表结构

```sql
CREATE TABLE `uni_device_trust` (
  `id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `device_id` VARCHAR(128) NOT NULL COMMENT '设备唯一标识',
  `username` VARCHAR(128) NOT NULL COMMENT '绑定的用户名',
  `computer_name` VARCHAR(64) DEFAULT NULL COMMENT '计算机名',
  `os` VARCHAR(32) DEFAULT NULL COMMENT '操作系统',
  `os_version` VARCHAR(64) DEFAULT NULL COMMENT '系统版本',
  `app_version` VARCHAR(16) DEFAULT NULL COMMENT '应用版本',
  `trust_level` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '信任级别: 0=未验证 1=已验证 2=信任',
  `last_active_time` DATETIME DEFAULT NULL COMMENT '最后活跃时间',
  `create_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE INDEX `uk_device_id` (`device_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='设备信任表';
```

### 9.3 服务设计

```java
public interface DeviceTrustService {
    /**
     * 记录设备活跃
     * 首次访问时创建，后续更新 last_active_time
     */
    void recordDeviceActive(String deviceId, String username, DeviceInfo info);
    
    /**
     * 获取设备信息
     */
    DeviceTrust getByDeviceId(String deviceId);
}
```

当前实现规则：
- 桌面端认证时提交 `deviceId`、`computerName`、`os`、`osVersion`、`appVersion`
- 如果 `username + deviceId` 已存在并且状态为 `trusted`，更新最后活跃时间并继续签发 Token
- 如果设备不在信任列表，后端创建 6 位验证码和 `verificationId`
- 测试阶段只把验证码写入后端日志；真实邮件发送位置保留 TODO
- 用户输入验证码后，后端把该设备写入 `uni_device_trust` 并签发 Token
- 认证过程写入 `uni_auth_audit`，人员信息写入 `uni_user_snapshot`

### 9.4 Token 刷新

`POST /api/v1/auth/token/refresh` 用于刷新当前 Token 活跃时间。该接口需要携带现有 Sa-Token，成功后返回当前 token 和前端本地 Session 建议过期时间。

## 十、前端状态机

### 10.1 AuthStatus 状态

| 状态 | 说明 | UI 展示 |
|------|------|---------|
| `checking` | 正在验证 Windows 用户 | 加载动画 |
| `verified` | 认证成功 | 正常显示应用 |
| `rejected` | 认证失败（用户未授权/停用） | 错误页面 |
| `offline` | 无法连接认证服务 | 离线提示 |

### 10.2 AuthError 错误详情

```typescript
interface AuthError {
  code: 'USER_NOT_FOUND' | 'USER_DISABLED' | 'DEVICE_NOT_TRUSTED' | 'SERVICE_UNAVAILABLE' | 'NETWORK_ERROR' | 'TOKEN_INVALID';
  message: string;
  detail?: string;
}
```

### 10.3 认证流程状态图

```
[App Start]
    │
    ▼
[authStatus = 'checking']
    │
    ▼
getCurrentWindowsUser() + getDeviceInfo()
    │
    ▼
verifyDesktopApi(request)
    │
    ├─── 成功 ──→ [authStatus = 'verified', 保存 currentUser + accessToken]
    │
    ├─── 401 用户不存在 ──→ [authStatus = 'rejected', errorCode = 'USER_NOT_FOUND']
    │
    ├─── 403 用户停用 ──→ [authStatus = 'rejected', errorCode = 'USER_DISABLED']
    │
    ├─── 网络错误 ──→ [authStatus = 'offline', errorCode = 'NETWORK_ERROR']
    │
    └─── 服务错误 ──→ [authStatus = 'rejected', errorCode = 'SERVICE_UNAVAILABLE']
```

## 十一、异常场景处理

### 11.1 认证失败

| 场景 | 处理 |
|------|------|
| 用户不在公司系统 | 显示"当前 Windows 用户未开通 UniComm 权限" |
| 用户已停用 | 显示"您的账号已被停用，请联系管理员" |
| 服务不可用 | 显示"认证服务暂时不可用，请稍后重试" |
| 网络超时 | 显示"网络连接超时，正在重试..." |

### 11.2 Token 失效

| 场景 | 处理 |
|------|------|
| Token 过期 | 清除本地 Session，重新触发认证流程 |
| Token 无效 (401/403) | 清除本地 Session，重新触发认证流程 |

**注意：** 不是显示登录页，而是重新执行桌面端认证流程。

### 11.3 离线处理

| 场景 | 处理 |
|------|------|
| 无法连接服务器 | 进入离线模式，显示本地缓存数据 |
| 离线期间操作 | 存入本地操作队列，恢复后同步 |
| 恢复网络 | 自动重连，重新认证，继续操作 |

## 十二、安全原则

| 原则 | 说明 |
|------|------|
| 后端校验 | 前端传来的 username 不能直接信任 |
| 服务端签发 | Token 必须由后端签发 |
| 数据隔离 | Memo 数据按 owner_username 隔离 |
| 日志安全 | accessToken 不允许打印到日志 |
| 临时数据 | 临时用户仅用于 Phase 1 联调 |
| 真实环境 | 必须接入公司人员接口，并关闭 mock 人员源 |
